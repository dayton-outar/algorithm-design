# Coding Lessons

Most algorithm problems don’t reward brute force—they reward clarity of thought.

Early on, it’s easy to assume that strong problem solvers rely on flashes of brilliance. In reality, what separates them is something much more repeatable: they compress problems into smaller, simpler pieces and extract patterns from them.

When faced with a problem, the instinct might be to jump straight into coding. That’s usually the wrong move. The better approach is to slow down and ask:

* What happens in the smallest possible cases?
* Can larger cases be expressed as combinations of smaller ones?
* Is there a constraint or invariant that must always hold?

Strong problem solvers naturally reduce complexity before they ever write a line of code. They test tiny inputs, observe behavior, and look for structure. Once a pattern appears, they generalize it. That generalization is where efficiency comes from.

For example, instead of evaluating every possible scenario, they ask whether solving a limited subset is enough. Often, it is.

This repository is built around that mindset.

---

## The Working Method

When tackling problems here, the goal is not just to arrive at a correct solution, but to build a repeatable thinking process:

1. **Start Small**
   Ignore the full problem size. Work with the smallest inputs where you can see everything clearly.

2. **Experiment Actively**
   Try concrete examples. Break them. Compare outcomes. Let the behavior of the problem reveal structure.

3. **Look for Invariants**
   Ask what must always be true, regardless of input size. These constraints are often the key to simplification.

4. **Reduce the Problem**
   See if larger cases can be constructed from smaller ones. If they can, you may not need to solve the large case directly.

5. **Generalize the Pattern**
   Once something works for small inputs, expand it carefully. This step turns observation into an algorithm.

## What This Repo Is Really About

This is not a collection of answers.

It’s a record of thinking—how problems are broken down, how patterns are discovered, and how efficient solutions emerge from that process.

The goal is simple:

> Build the habit of seeing structure where others see complexity.

If you can do that consistently, the “genius moments” stop being rare.
